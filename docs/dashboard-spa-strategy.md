# ダッシュボード SPA 再設計戦略

## 背景

現状の `dashboard` 画面には、以下の異なる目的の UI が同居している。

- 今週の出席入力
- 過去週を含む出席確認と編集
- 生徒登録
- 教師情報の表示

PC では 1 画面に収まっていても、スマホでは以下の問題が大きい。

- 1 つの作業に集中しづらい
- 横幅を使う表 UI が見づらい
- 重要度の低い情報が主操作の近くにあり、判断コストが高い
- ページ上での移動量が多く、毎週の入力が遅い

このため、単純なページ分割よりも、`/dashboard` を中心にした SPA として再設計する。

## 設計方針

### 基本方針

- ルートは原則 `dashboard` に集約する
- 画面内で目的別ビューを切り替える SPA 構成にする
- スマホ利用を最優先にする
- 最重要操作である「今週の出席入力」を初期表示にする
- URL クエリで状態を同期し、戻る操作と共有性を維持する

### 重要度の優先順位

1. 今週の入力を素早く終えられること
2. 出席全体を確認、修正できること
3. 生徒登録ができること

この優先順位に合わせて、情報設計と UI の主従を決める。

## 目標 UX

### 目標 1: 今週の入力を最短手数で終える

- ログイン後は自動で「今週」タブを開く
- 日曜であれば当日の日付を初期選択する
- 担当クラスが 1 つなら自動選択する
- 担当クラスが複数なら、最初にクラスだけ選べば入力を始められる
- 生徒ごとの状態変更を 1 タップまたは 2 タップ以内で完了できる
- 保存操作は常に視界内にある

### 目標 2: 管理用の確認操作を主画面から分離する

- 過去週の確認や編集は「出席」タブに分離する
- 生徒登録は「生徒」タブに分離する
- 教師情報のような補助情報は主画面から外すか縮小する

### 目標 3: テーブル依存を減らす

- スマホでは表よりカード UI を優先する
- 横スクロールが必須な UI は避ける

## 情報設計

### タブ構成

`dashboard` 内に以下の 3 タブを持つ。

1. `今週`
2. `出席`
3. `生徒`

初期表示は `今週` とする。

### URL 状態

以下のようなクエリを使い、SPA 内状態を URL と同期する。

- `tab=week | attendance | students`
- `classId=<class-id>`
- `date=<yyyy-mm-dd>`

例:

- `/dashboard?tab=week&classId=xxx`
- `/dashboard?tab=attendance&classId=xxx&date=2026-04-05`

## 各タブの役割

### 1. 今週

最優先の操作画面。考えるための画面ではなく、入力を終わらせるための画面にする。

#### 表示内容

- クラス選択
- 対象週の日付表示
- 入力状況サマリー
- 生徒ごとの出席入力カード
- 保存アクション

#### UI 方針

- 生徒一覧はテーブルではなく縦カード
- 1 カードに以下を収める
  - 生徒名
  - 学年
  - 出席状態トグル
  - 任意メモ
- 出席状態は大きい 2 択 UI を優先する
- 保存ボタンは画面下部固定バーに置く
- 入力済みデータがある場合は「再編集」であることを明示する

#### あえて載せないもの

- 年度全体の日曜一覧
- 生徒登録フォーム
- 教師プロフィール詳細

### 2. 出席

今週以外を含む、出席の確認と再編集のための画面。

#### 表示内容

- クラス選択
- 対象週の選択
- 週ごとの入力状況一覧
- 選択週の出席編集

#### UI 方針

- 日曜一覧はカードまたはコンパクトなリストにする
- 各週に「入力済み人数」「出席数」「欠席数」を表示する
- 週選択後は、その週の出席カード一覧を表示する

### 3. 生徒

生徒登録と一覧確認のための画面。出席操作と分離する。

#### 表示内容

- 対象クラスの生徒一覧
- 新規生徒登録フォーム

#### 将来拡張

- 生徒編集
- 生徒の無効化
- クラス移動

## レイアウト方針

### モバイル優先

- 上部に現在のクラス選択
- 中央にタブ別コンテンツ
- 下部に固定タブナビ

### 補助情報の扱い

- 教師メールや権限表示は常時大きく見せない
- 必要であればヘッダー内の小さな情報へ縮小する
- 重要メッセージだけを優先表示する

## 状態管理の考え方

### サーバー起点

- クラス一覧
- 生徒一覧
- 出席履歴
- 当日データの初期値

これらはサーバー側で取得する。

### クライアント起点

- 現在タブ
- クラス切替
- 今週タブ内の入力中状態
- 保存前の変更差分

このため、`dashboard` の一部をクライアントコンポーネント化する前提で考える。

### Phase 1: 画面の責務分離

目的:

- 現在の 1 画面を、3 タブ相当の責務に分ける
- URL に `tab` を導入する
- 既存のサーバーアクションを維持したまま見た目の主従を整理する

やること:

- `dashboard` の UI を `今週 / 出席 / 生徒` に分割する
- 既存の教師情報カードを縮小または退避する
- 生徒登録をメイン画面から分離する
- 年度全体一覧を「今週」から外す

### Phase 2: 今週タブの高速化

目的:

- 毎週の入力を最短導線にする

やること:

- 今週タブをカード UI に変更する
- 出席状態をスマホで押しやすいトグルにする
- 下部固定の保存バーを追加する
- 入力済みサマリーを見える位置に出す

進捗:

- カード UI 化、2 択トグル、固定保存バー、入力済みサマリーまで完了
- まだクライアント状態は持たず、server action ベースの form 送信を維持している

### Phase 3: 出席タブの再設計

目的:

- 過去週の確認と再編集をわかりやすくする

やること:

- 日曜一覧をテーブルからリスト中心へ変更する
- 週選択後の編集 UI を今週タブと共通化する
- 履歴表示と編集導線を整理する

進捗:

- 日曜一覧のカード型リスト化は完了
- 週選択後の編集 UI は今週タブと共通化済み
- 週切替自体はまだ通常の URL 遷移ベース

### Phase 4: 生徒タブの整備

目的:

- 登録作業を独立させる

やること:

- 生徒一覧と追加フォームを生徒タブへ集約する
- クラス文脈を維持したまま追加できるようにする

## コンポーネント分割の方向

- `DashboardShell`
- `DashboardTabNav`
- `ClassSwitcher`
- `TodayAttendancePanel`
- `AttendanceHistoryPanel`
- `StudentManagementPanel`
- `AttendanceStudentCard`
- `StickySaveBar`

ポイント:

- 出席入力部分は、今週タブと出席タブで再利用できるようにする
- サーバー取得データとクライアント操作状態の境界を明確にする

## 捨てる判断

以下は優先的に捨てる、または後ろへ下げる。

- PC 的な 2 カラム前提の密な情報配置
- 出席入力と生徒登録の同居
- スマホで見づらい横長テーブル
- 主操作の近くにある補助情報カード

## 成功条件

- ログイン直後に「今週の入力」に迷わず入れる
- スマホで 1 クラス分の入力を無理なく終えられる
- 過去週の確認と編集先が直感的にわかる
- 生徒登録が主操作を邪魔しない

## 次に決めること

- 「今週」の日付判定をどうするか
  - 日曜のみ当日を使うか
  - 平日は直近または次回の日曜へ寄せるか
- 出席状態を MVP では `出席 / 欠席` に寄せるか
- 今週タブの保存を即時保存にするか、まとめて保存にするか
- クラス切替を上部固定にするか、ヘッダー内に収めるか
